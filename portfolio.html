왜 점진적 전환(Strangler Fig)이 아닌 부분적 빅뱅을 선택했는가?
아파트 도메인과 매칭 도메인이 양방향으로 강결합 — 아파트 매물 등록 시 매칭 로직이 실행되고, 매칭 결과가 아파트 상태를 변경하는 구조. 하나만 분리하면 나머지가 작동하지 않았습니다. 교과서적인 점진적 전환은 이 구조에서 불가능했고, 팀 규모(2~3명)가 감당할 수 있는 범위 내에서 최소한의 빅뱅을 설계했습니다.
팀 내 합의 과정: CTO와 기술 논의를 거쳐 "강결합 도메인은 동시 분리, 나머지는 점진적 전환"이라는 하이브리드 전략에 합의했습니다. 동료 개발자들과 함께 헥사고날 아키텍처의 Port/Adapter 경계를 정의하고, 코드 리뷰를 통해 도메인 로직이 인프라에 의존하지 않는지 지속 검증했습니다.

  • ① Bean 이름 충돌 (ConflictingBeanDefinitionException)
    원인: 12개 서비스에 각각 SecurityConfig, WebConfig 등 동명 클래스 존재 → Spring의 flat bean registry는 패키지 무관, 클래스명으로 등록 → 단일 ApplicationContext에서 충돌
    해결: 26개 파일에 @Component(name="module-xxx") 명시적 이름 지정
  • ② 순환 의존성 폭발
    원인: FeignClient는 HTTP 호출이라 A↔B 순환 호출이 런타임에 가능 → ModuleApi(메서드 호출)로 전환하면 컴파일 타임 순환 참조로 변환 → Spring 컨텍스트 기동 실패
    해결: @Lazy 프록시로 지연 초기화 50곳 적용 — 장기적으로 의존 방향 정리 예정
  • ③ @Transactional 사일런트 버그
    원인: Schema-per-Service로 서비스별 TM 존재 → 앱 통합 시 @Primary TM이 모든 bare @Transactional 흡수 → agent의 트랜잭션이 apart의 TM으로 실행

  • 11개 Schema·EntityManager·TM 분리 유지 — 필요 시 다시 MSA로 분리 가능
왜 ③이 가장 위험한가:
각 서비스는 자기 Schema 전용 TransactionManager를 갖고 있었습니다. 앱 통합 후 @Primary로 지정된 하나의 TM이 명시적 TM이 없는 모든 @Transactional을 가져갑니다. 결과: agent 도메인의 쿼리가 apart의 DB 커넥션으로 실행 → 커밋/롤백 범위가 달라져 데이터 정합성이 깨짐. 단위 테스트에서는 단일 TM 환경이라 재현 불가 — 통합 테스트 + 실제 다중 DataSource 환경에서만 발견 가능.
MSA에서 되돌아가는 것이 아니다: 모듈 경계와 Schema-per-Service는 그대로 유지. 배포 단위만 통합하여 2명이 운영 가능한 현실적 아키텍처로 전환. 필요 시 다시 분리 가능.

Isolation Level을 낮추는 것이 올바른 판단이었는가?
Spring Batch의 Metadata 테이블은 Job 실행 상태를 기록하는 용도입니다. 동시 Job 실행 시 같은 row를 경쟁하지만, 최종 상태만 정확하면 중간 격리는 불필요합니다. 이 판단의 근거를 팀에 공유하고, 실제 병렬 실행 테스트로 데이터 정합성을 검증한 후 적용했습니다.
Tasklet→Chunk 전환의 숨은 이유: MVCC 데이터 누락
Tasklet 방식은 전체 작업이 하나의 트랜잭션 — MVCC 스냅샷으로 배치 실행 중 입금된 데이터가 누락되는 문제 발견. Chunk 방식으로 전환하여 커밋 단위를 분리하고, 최신 데이터를 안정적으로 처리할 수 있게 개선했습니다.
